# 舆情报告生成模块 — 开发任务书

> 本文档面向报告模块开发者。你将构建一个**配置驱动的舆情分析报告生成引擎**。
> 仓库中不包含任何生产环境凭据；所有外部依赖（数据库、大模型）一律通过 `.env` 注入。

---

## 一、任务背景与目标

我们的舆情监测系统持续采集多平台关于监测对象的文章，每篇文章已带有情感分类（positive/neutral/negative）、
负面程度（1–5）、严重性分级、摘要、互动数据等结构化字段。

系统的 Web 端有一个报告页面（报告构建器），包含两个标签页：

- **已生成报告**：展示历史报告 PDF 列表（读取 `index.json`），已上线。
- **配置新报告**：用户选择报告类型、语言、话题、时间范围和章节，点击生成 —— **目前只有
  界面，没有完善的后端。你的任务就是实现这个"生成"**。

最终形态：用户在页面上提交配置 → 你的引擎查询数据库、生成图表、由大模型撰写各章节
叙述、组装成 PDF → 报告出现在"已生成报告"列表中。

**分阶段交付**：先做成命令行工具（M1、M2），再包装成 API 服务（M3）。上线部署由
Blair 负责，你暂时不需要接触生产环境。

## 二、两个固定契约（不可更改的接口）

引擎内部架构由你决定，但输入和输出两端是固定的：

### 2.1 输入：`report-config.json`

前端"保存"按钮导出的配置文件，即引擎的唯一输入（样例见 `examples/`）：

```jsonc
{
  "reportType": "csuite",        // "csuite"（决策版）| "pr"（公关版）
  "language": "zh",              // "zh" | "en"
  "topic": {
    "tag": "layoff",             // articles.tags 中的话题标签
    "displayName": "裁员",        // 话题中文名
    "eventTitle": "××\"裁员60%\"事件"  // 报告标题前缀
  },
  "dateRange": { "from": "2026-03-17", "to": "2026-03-23" },
  "sections": [                  // 19 个章节，按数组顺序渲染
    { "id": "verdict", "enabled": true },
    { "id": "response", "enabled": false, "input": { "responseDate": null } },
    { "id": "benchmark", "enabled": false, "input": { "comparisonTag": null } },
    { "id": "biz-impact", "enabled": false, "input": { "notes": null } }
    // ... 其余章节
  ]
}
```

19 个章节 ID 及内容要求见 `docs/02-report-spec.md`。未知的 `reportType` 按 csuite 处理。

### 2.2 输出：报告包（bundle）目录

```
out/{id}/
├── report.md        # 完整报告 markdown
├── report.pdf       # A4 排版 PDF（样式参照 examples/gold-report/）
├── charts/*.png     # 所有图表（150 dpi，中文字体正常渲染）
└── meta.json        # 元数据，字段如下
```

```jsonc
// meta.json —— 与前端 ReportMeta 类型一一对应
{
  "id": "layoff-2026-03-23",
  "title": "××\"裁员60%\"事件舆情分析报告",
  "reportType": "csuite",
  "language": "zh",
  "topic": "裁员",
  "dateRange": { "from": "2026-03-17", "to": "2026-03-23" },
  "sections": 7,                 // 实际渲染的章节数
  "charts": 8,                   // 图表数
  "stats": { "articles": 171, "negativeRatio": "81.8%", "peakDay": "3/21" },
  "file": "/reports/layoff-2026-03-23.pdf",
  "generatedAt": "2026-07-15T10:00:00+08:00"
}
```

## 三、核心设计要求

1. **流水线 + 每章节一次 LLM 调用**。对每个启用的章节：固定 SQL 取数 → 代码生成图表 →
   将查询结果交给大模型撰写该章节叙述（markdown 片段）→ 最后统一组装。整份报告的
   LLM 调用次数 ≤ 章节数（≤19 次），成本可预算。
2. **所有数字由代码计算，LLM 只写文字**。占比、峰值、总量等统计一律在 SQL/Python 中算好
   再注入提示词；严禁让模型自己算数或编造数字。报告中的每个数字必须能追溯到查询结果。
3. **观点必须来自数据**。"主要观点"等章节的主题归纳必须基于文章摘要字段（summary），
   引用或转述真实标题/摘要，不得泛泛而谈。
4. **章节级容错**。某一章节失败（SQL 空结果、LLM 超时）→ 该章节标注缺失、其余章节照常，
   整份报告不许因单章节失败而中断；失败信息记入 `meta.json`。LLM 调用需要重试与退避。
5. **配置全部走环境变量**（`.env`，参照 `.env.example`）：
   - `PG_DSN` —— PostgreSQL 连接串（开发期指向本地 fixtures 库；未来切换数据源时只改这一行，代码零改动）
   - `LLM_BASE_URL` / `LLM_API_KEY` / `LLM_MODEL` —— 任意 OpenAI 兼容接口，**模型你自选**
   - 仓库中永远不允许出现任何真实密钥、密码、内网地址。
6. **PDF 渲染跨平台**（建议 Playwright 自带 Chromium 或 WeasyPrint，不依赖本机装了什么浏览器）。
   A4、中文字体、分页规则参照 `examples/gold-report/report.html` 中的 CSS。
7. **图表风格统一**：白底、隐藏上/右边框、150 dpi、负面 #DC2626 / 中性 #F59E0B /
   正面 #10B981；标题写洞察（如"负面占比高达81.8%"）而非"情感分布图"。

## 四、参考架构（仅供参考，可推翻）

以下是一种可行拆分，你可以改造或重来，只要第二节的契约不变：

```python
class Section:
    id: str                                        # 与前端章节 id 一致
    def fetch(self, db, scope) -> SectionData      # 固定 SQL，按 scope 参数化
    def charts(self, data, outdir) -> list[Path]   # matplotlib 出图
    def narrate(self, llm, data, ctx) -> str       # 一次 LLM 调用 → markdown 片段
```

`scope` = 话题过滤条件（tag）+ 时间范围 + 品牌名，由配置构建一次、全局共享。
报告类型（csuite/pr）与语言（zh/en）应当只是提示词模板的参数，而不是两套代码。

## 五、开发环境（30 分钟内跑通）

1. `cd fixtures && docker compose up -d` —— 启动本地 PostgreSQL 并自动导入样例数据
   （真实历史话题的文章数据，含情感、摘要、互动字段）。
2. `cp .env.example .env`，填入本地 DSN 和你自选的大模型接口信息。
3. 验证：对 fixtures 库执行 `docs/02-report-spec.md` 中任意一条 SQL，能返回数据即环境就绪。
4. 参照 `examples/gold-report/` 阅读一份真实报告成品 —— 这是质量基准线。

## 六、里程碑与验收标准

### M1 —— 复现现有报告（中文）
`report generate --config examples/report-config.csuite.json --out ./out/` 一条命令产出
完整报告包。验收：
- csuite（7 章节）与 pr（11 章节）两种默认配置均可生成；
- markdown 中每个数字与直接对 fixtures 执行 SQL 的结果一致；
- 图表中文无乱码；PDF 分页、样式与 gold-report 相当；
- 每章节 LLM 调用 ≤1 次；单章节失败不中断整体。

### M2 —— 12 个新章节 + 英文
实现全部 19 个章节（含 3 个带用户输入的章节：回应效果/历史对标/商业影响），支持
`language: "en"`。验收：任意章节组合的配置均能生成结构正确的报告。

### M3 —— API 服务
FastAPI：`POST /reports`（body 为 ReportConfig）→ 返回任务 id；
`GET /reports/{id}/status`；完成后可下载报告包。进程内任务队列即可。
验收：并发提交 2 个任务不互相干扰；服务重启后已完成报告仍可下载。

### 上线（目前由Blair 负责，不在你的范围）
数据源切换、服务部署、前端对接、报告发布。

## 七、工作约定

- 小步提交，提交信息说清楚做了什么；主分支保持随时可运行。
- 测试：SQL 层对 fixtures 做集成测试；LLM 层用可注入的假模型（stub）做确定性测试，
  真实模型只做冒烟验证。
- 任何时候发现文档与代码/数据不符，先在 issue/沟通中提出，不要静默绕过。
- 遇到规范没写清楚的地方：先按你的判断实现，并在 PR 描述中标注该决定，便于讨论。
